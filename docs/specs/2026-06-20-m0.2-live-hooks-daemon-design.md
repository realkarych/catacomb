# Catacomb M0.2 — Live Hooks Daemon — Design

- **Status:** Accepted (scoping approved)
- **Date:** 2026-06-20
- **Deciders:** @realkarych
- **Builds on:** M0.1 (offline core & replay, merged). Parent design: `docs/specs/2026-06-20-catacomb-design.md`. Related ADRs: 0001 (daemon form), 0002 (four-source/hooks backbone), 0010 (observation identity & durability), 0011 (execution-scope ids), 0012 (run lifecycle), 0013 (daemon security), 0014 (status reconciliation), 0017 (data-format versioning), 0018 (time), 0019 (operability).

## Goal

Make Catacomb go **live**: a long-running daemon receives Claude Code **hook** events in real time, turns them into the same canonical execution graph M0.1 already builds offline, and persists it durably as it happens. After `catacomb install-hooks`, a normal Claude Code session is traced into SQLite with no further action.

## Scope

This is the **tight go-live slice**. Hooks are the only live source; the offline JSONL `replay` from M0.1 is unchanged.

**In scope (M0.2 v1):**

1. A `catacomb daemon` long-running process serving HTTP/1.1 over a `0600` unix-domain socket.
2. A `catacomb hook <type>` forwarder (same binary) that reads the hook event JSON on stdin and POSTs it to the daemon, **failing open**.
3. `catacomb install-hooks [--project|--global] [--uninstall]` that wires the forwarder into Claude Code `settings.json`.
4. An `ingest/hook` mapper turning each of the **nine** hook events into `model.Observation`s.
5. A live ingest pipeline: per-observation **atomic** append-and-apply, a **persisted `seq` watermark**, and **boot recovery** (ADR-0010 clauses 3, 5, 6).
6. Minimal **run lifecycle** (`SessionStart` opens, `Stop`/`SessionEnd` closes) backed by a `runs` table (ADR-0012, minimal subset).
7. Minimal **status** from hooks: `PreToolUse`→`running`/`blocked`, `PostToolUse`→`ok`/`error`, with genuine-terminal-beats-provisional precedence (ADR-0014 clause 2, minimal subset).

**Deferred to M0.3:** idle-reaper + `abandoned`/`unknown` statuses; full status-lattice supersede/cancel cascade; full ADR-0019 operability surface (health endpoint, queue-depth/hook-drop metrics, per-adapter supervision, `lossy` flagging); TCP ingress + bearer token (ADR-0013 clause 2).

**Deferred to M1/M2 (unchanged from the parent spec):** OTLP receiver + four-source precedence (M1); stream-json reader + live JSONL fsnotify tailer + subagent-tree backfill + markers (M2); realtime WS/SSE/gRPC surfaces (M3); neo4j/postgres exporters (M4); web UI (M5).

**Idempotent re-ingest (ADR-0010 clause 4) is deferred to the second live source (M2 tailer).** Hooks fire once; crash-safety in v1 comes from atomic apply + watermark + log-replay recovery (which re-derives in-memory state without re-INSERTing observations). A cross-source dedup key is only needed once two live sources can report the same event.

## Architecture

New packages keep M0.1's boundaries: core packages (`model`, `ingest/*`, `reduce`, `store`, `export/*`) import no daemon/CLI/network code.

- **`ingest/hook/`** — pure mapper. `Parse(hookType string, payload []byte, executionID string, nextSeq func() uint64) ([]model.Observation, error)`. Parallel to `ingest/jsonl`: defensive, version-tolerant decoding (parent spec §17), committed per-event fixtures that ARE the contract. No network/daemon dependencies.
- **`daemon/`** — the long-running server. Owns: the `net/http` server on a unix listener; the route table (`POST /hook/<type>`, plus a stub `GET /healthz` returning 200 — the full operability surface is M0.3); the in-memory live `reduce.Graph` per execution; the run-lifecycle bookkeeping; boot recovery. Imports core + `ingest/hook`. No `cmd`/cobra dependency.
- **`cmd/catacomb/`** — new subcommands `daemon`, `hook`, `install-hooks` (thin wrappers over testable functions, like M0.1's `replay`).
- **`store/`** (extended) — live atomic append-apply, persisted seq counter + watermark read, observation read-since for recovery, and a minimal `runs` table. The batch `Persist` is retained for `replay`.

### Architectural choice (recorded)

The daemon holds the live `reduce.Graph` **in memory** per execution and flushes node/edge deltas to SQLite **per observation in one transaction**; on boot it rebuilds memory by replaying observations with `seq > watermark`. The rejected alternative — a stateless daemon that re-reduces from the log on every read — was dropped: the reducer is inherently stateful, and re-reduction is wasteful with no realtime query surface until M3. The in-memory graph reuses M0.1's `reduce.Graph`/`Apply` unchanged in structure.

## Data flow

```
Claude Code → runs hook command:  catacomb hook PostToolUse  < event.json
  forwarder: read stdin → POST http+unix://<socket> /hook/PostToolUse  (body = raw event JSON)
             FAIL-OPEN: daemon down / any error → warn to stderr, exit 0, never block the agent
  daemon receives → ingest/hook.Parse(type, payload, executionID, nextSeq) → []model.Observation
                    (ULID obs_id, daemon-assigned seq, execution_id, correlation, event_time)
  for each observation → ONE sqlite transaction:
       INSERT observation ; reduce.Apply(obs) → node/edge upserts ; advance persisted seq watermark
  update the in-memory reduce.Graph for that execution
  boot recovery: read watermark → ObservationsSince(watermark) → reducer rebuild (no re-INSERT)
```

`export/jsonl.Snapshot` and offline `catacomb replay` keep working unchanged against the same store.

## Transport and socket

- HTTP/1.1 over a unix-domain socket with `0600` permissions (owner-only) — the filesystem permission IS the trust boundary for v1 (ADR-0013); no bearer token (that is the TCP path, deferred).
- Socket path resolution order: `CATACOMB_SOCKET` env override → `$XDG_RUNTIME_DIR/catacomb/daemon.sock` → `~/.catacomb/run/daemon.sock`. The daemon creates the parent dir (`0700`) and the socket (`0600`), removing a stale socket on start.
- The forwarder uses `http.Client` with a unix dialer. It is dependency-light and **fails open**: connection refused / timeout / non-2xx → one-line stderr warning, exit 0. It never blocks or fails the agent.
- `install-hooks` writes the resolved socket path into the hook command (as a `CATACOMB_SOCKET=<path>` prefix) so forwarder and daemon agree without ambient discovery.

## Hook events → observation mapping

All nine events are accepted. Structural events produce graph nodes/edges/status; the rest are recorded as observations carrying a marker (no structural node in v1). `tool_use_id` is the cross-source linchpin (parent spec §5.5); `session_id` keys the execution; `agent_id` keys subagent nodes.

| Hook event | Produces |
| --- | --- |
| `SessionStart` | open `session` node + `runs` row (status `running`); mint `execution_id` for this `session_id` |
| `UserPromptSubmit` | `user_prompt` node (keyed by `uuid` if present, else event time); edge session→prompt |
| `PreToolUse` | `tool_call`/`mcp_call` node (keyed by `tool_use_id`), status `running` (or `blocked` if permission denied in payload); edge turn/session→tool |
| `PostToolUse` | close the same `tool_call`/`mcp_call` (status `ok`/`error` from `is_error`/`tool_response`); attach output payload + hash |
| `SubagentStop` | close `subagent` node (keyed by `agent_id`); attach `agent_transcript_path` if present |
| `Stop` | session quiescence signal → mark session ended (note: does not fire on user interrupt, parent spec §5.8) |
| `SessionEnd` | `session_ended` → close `runs` row (status `ok`), set session node `t_end` |
| `PreCompact` | observation + marker (`compact_boundary`); no structural node in v1 |
| `Notification` | observation + marker; no structural node in v1 |

Field-to-`Correlation`/`Attrs` mapping is pinned by the committed fixtures. Unknown event types and unknown fields are tolerated (logged, skipped/ignored) rather than fatal (parent spec §17).

## Identity, time, sequence

- **`execution_id`** is minted **per live session**, keyed by `session_id`, on first sight (typically `SessionStart`), persisted in `runs`, and reused for every observation of that session. (M0.1 minted one `execution_id` per `replay` invocation; the canonical-id helpers from `model` are otherwise unchanged — ADR-0011.)
- **`seq`** is a daemon-owned **persisted, global, monotonic** counter (`store.NextSeq()` backed by a counter row), stamped at the observation-append boundary (ADR-0010 clause 1, ADR-0018). The **watermark** is the max `seq` whose observation is committed-and-applied.
- **`event_time`** comes from the hook payload timestamp when present, else the daemon receive time; `observed_at` is the daemon receive time (ADR-0018, time model: `seq` is the order).

## Store changes

- New `runs` table: `(execution_id PK, session_id, started_at, ended_at, status, meta)`.
- New persisted counter row for `seq` (and the watermark, which equals the max committed `seq`).
- New API on `Store` (additive; batch `Persist` retained for `replay`):
  - `NextSeq() (uint64, error)` — allocate the next global seq.
  - `AppendAndApply(obs model.Observation, nodes []*model.Node, edges []*model.Edge) error` — one transaction: INSERT the observation, upsert the node/edge deltas, advance the watermark.
  - `Watermark() (uint64, error)` — highest committed-and-applied seq.
  - `ObservationsSince(seq uint64) ([]model.Observation, error)` — ordered replay feed for recovery.
  - `OpenRun` / `CloseRun` (or an upsert) for run lifecycle rows.
- WAL stays on (M0.1). Observations remain INSERT-only (ADR-0010 clauses 1–2).

## Status (minimal lattice)

`reduce` gains a small status-precedence rule: a **genuine terminal** status from a real event (`ok`/`error` from `PostToolUse`, `blocked` from a denied `PreToolUse`) outranks a **provisional** status (`running`/`pending`), regardless of arrival order (ADR-0014 clause 2). This must not regress M0.1's JSONL path, where `tool_result` already sets `ok`/`error` directly. The supersede/cancel **cascade** (ADR-0014 clause 3) and `unknown`/`abandoned`/`superseded` statuses are deferred to M0.3.

## Run lifecycle (minimal)

`SessionStart` opens a run (`runs` row + `session` node, status `running`). `SessionEnd` closes it (status `ok`, `t_end` set); `Stop` marks session quiescence. No idle-reaper, no `abandoned`/eviction-gate logic in v1 — a session that never emits `SessionEnd` simply stays `running` until M0.3 adds the reaper.

## install-hooks

- Default scope **`--project`**: writes `.claude/settings.json` in the current directory. `--global` targets `~/.claude/settings.json`.
- Writes Claude Code hook entries that run `CATACOMB_SOCKET=<path> catacomb hook <type>` for each of the nine event types.
- **Idempotent merge:** existing unrelated hooks are preserved; re-running is safe (catacomb's own entries are detected and not duplicated).
- **`--uninstall`** removes only catacomb's entries, leaving other hooks intact.
- If the target `settings.json` is malformed, fail with a clear message rather than clobbering it.

## Error handling and trust boundary

- Forwarder fails open (never blocks the agent).
- Daemon tolerates malformed/unknown hook payloads: log and skip the event; never crash the server.
- A per-event transaction failure rolls back that event and is logged; the daemon keeps serving.
- `0600` socket + `0700` parent dir is the trust boundary; no network exposure in v1.
- The full operability surface (ADR-0019: structured health, metrics, supervision, `lossy` flagging) is M0.3; v1 ships only a `GET /healthz` 200 stub.

## Testing strategy

Same gates as M0.1: TDD, no comments, 100% line coverage (`make cover`), `golangci-lint` clean, cross-platform CI.

- `ingest/hook` — table-driven over committed, secret-free per-event fixtures; assert observation shapes/correlation/status mapping for all nine events plus malformed/unknown-type cases.
- `store` live API — temp-file SQLite: atomic append-apply, `NextSeq` monotonicity, watermark advance, `ObservationsSince` recovery feed, `runs` open/close; reuse the M0.1 opener/marshal seams for error paths.
- `daemon` — in-process `http.Handler` tests POST each event type and assert the resulting store/graph state; recovery test (restart over a populated store rebuilds the in-memory graph); malformed-payload tolerance.
- forwarder (`cmd`) — stdin→POST happy path against a test server; daemon-down → exit 0 + stderr warning (fail-open).
- `install-hooks` — temp HOME / temp project dir: fresh write, idempotent re-run, merge with pre-existing unrelated hooks, `--uninstall` removes only catacomb entries, malformed-settings refusal.
- End-to-end smoke — start the daemon on a temp socket, POST a scripted nine-event session, assert a golden live graph (session/prompt/tool/mcp/subagent nodes, statuses, edges), then `export` it.

## Risks / open items for the plan

- **Hook payload field names** are the undocumented surface (as JSONL was in M0.1). The committed fixtures are the contract; a one-time, never-committed check against a real `~/.claude` hook event confirms field names per event type.
- **Permission-denied detection** for `PreToolUse`→`blocked` depends on the hook payload's permission fields; the fixture pins the exact shape.
- **`runs` table vs reducer ownership** — the daemon owns run rows; the reducer owns nodes/edges. The plan must keep that boundary clean.
- **Per-event transaction cost** — one SQLite tx per hook event; acceptable at interactive rates, revisit batching only if a measured problem (YAGNI for v1).

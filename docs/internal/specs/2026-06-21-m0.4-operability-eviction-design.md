# M0.4 — Operability & memory-bounding (design)

- **Status:** Approved (autonomous design — see [[autonomous-completion-mandate]])
- **Date:** 2026-06-21
- **Deciders:** @realkarych (delegated; decisions documented here)
- **Builds on:** M0.3 (merged): finalization + live daemon + reaper + crash-isolation
- **Implements (subsets of):** ADR-0012 (eviction gated on liveness), ADR-0019 (operability/health/metrics, soft-caps)

## 1. Context & goal

M0.3's daemon never frees memory: `d.graphs` (one reducer shard per execution) and
`d.lastSeen` grow without bound — a terminal (`ok`/`abandoned`) run's shard stays
resident forever. And the daemon has only a `/healthz` liveness probe — no visibility
into open runs, shards, quarantines, or seq. M0.4 **bounds steady-state memory**
(reaper-driven shard eviction with lazy reload) and **makes the daemon observable**
(a `/metrics` endpoint), and clears small M0.1/M0.3 tech debt — completing the
hooks-only-buildable parts of ADR-0012/0019.

## 2. Scope

In scope:

- **Shard eviction + lazy reload** (ADR-0012): the reaper evicts a terminal run's
  in-memory shard after a cooldown; a later observation for an evicted execution
  lazily reloads its history from the store before applying.
- **Soft-cap** (ADR-0019 §5): a max-open-shards backstop that force-evicts the
  oldest-ended terminal shards when exceeded.
- **`GET /metrics`**: a JSON snapshot of the currently-measurable operational signals.
- **Store-write-error counter** feeding `/metrics`.
- **Tech-debt cleanup**: the M0.1 `nowFn`-override reduce test gap, the M0.1 JSONL
  timestamp-parse follow-up, the `var max` shadow rename, the `appendUnique`
  `slices.Contains` modernization.

Out of scope → later milestones (no producer/consumer exists yet):

- **`lossy`-run flag** → M1/M2 (real drop sources: exporter buffer drops, tailer gaps).
- **Dogfooding loop-break** → M1 (OTLP self-endpoint refusal) + M2 (tailer self-exclusion).
- **Adapter restart-with-backoff** → M1/M2 (the hook receiver is request-per-event;
  the only long-running adapters are the OTLP receiver and the tailer).
- **Exporter/tailer/OTel metrics** → land with those features.

## 3. Shard eviction + lazy reload (`daemon` + `store`)

**Invariant (ADR-0012):** the observation log is the system of record and is never
deleted in v1; eviction frees only the in-memory shard, which is fully recoverable
by replaying that execution's observations.

- **Store** gains `ObservationsForExecution(executionID string) ([]model.Observation, error)`
  (`SELECT body FROM observations WHERE execution_id = ? ORDER BY seq`; the
  `idx_observations_run_seq` index already exists — add an execution index if the
  plan finds it needed).
- **Eviction (reaper pass):** after `reapIdle` marks idle runs `abandoned`, a second
  pass evicts every shard whose run is **terminal** (`ok`/`abandoned`) and whose
  `EndedAt` is older than a **cooldown** (default = `reaperWindow`; invariant
  `cooldown ≥ reaperWindow` so a just-reaped run isn't immediately evicted, leaving a
  window for a genuine-terminal supersede). Eviction drops `d.graphs[execID]` and
  `d.lastSeen[runID]` but **keeps `d.execBySession`** (the lightweight
  `sessionID→execID` mapping) so a reawakening session maps back to the *same*
  execution. Evicted shards are counted (`/metrics`).
- **Lazy reload (`Ingest`):** when an observation's execution is in `execBySession`
  but missing from `d.graphs` (evicted), rebuild its shard from
  `ObservationsForExecution(execID)` before applying the new observation — so a
  reawakened run regains its full history (and the run reverts `abandoned`→`running`
  per the M0.3a reducer). A brand-new session (not in `execBySession`) starts a fresh
  shard as today.
- **Liveness gate:** never evict a shard whose run is still `running` (non-terminal).
- **Recover (v1 limitation, documented):** boot `Recover` still replays all
  observations into all shards, then steady-state eviction trims terminal ones. A
  selective Recover (rebuild only non-terminal runs from the runs table) is a future
  optimization, noted not built.

## 4. Soft-cap (`daemon`)

A `--max-shards` flag (default e.g. `4096`) bounds resident shards as a backstop to
the time-based reaper. When `len(d.graphs)` exceeds the cap, the reaper pass
force-evicts the **oldest-`EndedAt` terminal** shards first (never a `running` run)
until under the cap, logging+counting the action (`/metrics`). If all shards are
non-terminal and the cap is exceeded, log a warning and do not evict (correctness
over the cap — never drop a live run's graph).

## 5. `GET /metrics` (`daemon/server`)

A new **open** (loopback-only, no token — aggregate counts, no payloads; consistent
with `/healthz`) `GET /metrics` returning JSON:

```json
{
  "uptime_seconds": 0,
  "open_runs": 0,
  "shards": 0,
  "max_seq": 0,
  "quarantined": 0,
  "evicted": 0,
  "store_write_errors": 0,
  "reaper_window_seconds": 1800
}
```

Counts come from the daemon under `d.mu`: `open_runs` = runs with status `running`
across shards; `shards` = `len(d.graphs)`; `max_seq` = `d.seq`; `quarantined`,
`evicted`, `store_write_errors` = in-memory counters; `uptime_seconds` from a
`startedAt` stamped via `nowFn`. Decision: open (not token-gated) because the boundary
is loopback and the body is non-sensitive aggregate counts useful for a local
readiness/monitoring probe; token-gating is a trivial future change if needed.

## 6. Decomposition

Two plans:

- **M0.4a — memory-bounding** (`store` + `daemon`): `ObservationsForExecution`;
  eviction pass + cooldown + soft-cap; lazy reload on reawaken. Offline/`-race`
  testable (drive `reapIdle`/eviction directly with injected `nowFn`).
- **M0.4b — observability + tech debt** (`daemon/server` + `store`/`reduce`/`ingest`):
  `/metrics` endpoint + store-write-error counter; then the four tech-debt items.

## 7. Constraints (inherited, binding)

- Go 1.26 pure-Go; **no comments** in Go (`internal/codepolicy`); **100% line
  coverage** under `-race`; `golangci-lint v2` clean; never `go mod tidy`.
- Single-mutex daemon discipline (eviction + reload happen under `d.mu`); no
  `time.Sleep` (ticker/`require.Eventually`); time via `nowFn`; cross-platform.
- The observation log is the system of record; runs/shards are derived and
  rebuildable; eviction never deletes observations.

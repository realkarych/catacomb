## PR-N — Known stream-json system subtypes (F10)

**Mechanism.** `ingest/streamjson/streamjson.go:97-108`: the `build` case `"system"` recognizes only `init` (→ `session_start` partial) and `compact_boundary` (known-ignored); every other subtype returns `drift.Counts{ReasonUnknownSubtype: 1}` (`streamjson.go:108`), which the daemon aggregates and warns on — the V-1 `format drift … reason=unknown_subtype count=1100`. Grounded live (claude 2.1.199, `claude -p … --output-format stream-json --verbose`): the stream emits system subtypes `init`, `hook_started`, `hook_response`, `thinking_tokens`. The three non-`init` ones are shipping today and must be known-ignored, not drift.

**Fix.** Add `hook_started`, `hook_response`, `thinking_tokens` to a `knownIgnoredSystemSubtype` set beside `compact_boundary`; no observations emitted for them (hook events already arrive via the hook path; token counts arrive via assistant usage and `result`). Drift must still fire for a genuinely unknown subtype (the existing `warmup` test). No `ingest/jsonl` parity change: `ingest/jsonl/jsonl.go:167` accepts `type:"system"` lines wholesale without subtype discrimination. Detailed plan: [2026-07-04-pr-n-drift-subtypes.md](2026-07-04-pr-n-drift-subtypes.md).

## PR-Q — Cost rollup double-count on the stream-json path (F7)

**Mechanism.** `ingest/streamjson/streamjson.go:135-146` maps the terminal `result` event to an `assistant_turn` partial carrying the session-cumulative usage **and** the reported `total_cost_usd`. `reduce/reduce.go:67-72` keys it as `AssistantTurnID(execID, "")` (a `result` has no message id) — a separate pseudo-turn node — and `reduce/reduce.go:76-78` prices it: `pricing/pricing.go:49-50` short-circuits on `ReportedUSD` with `cost_source: "reported"` (stamped at `reduce/reduce.go:529`). Every real assistant turn is meanwhile priced `estimated` from its own usage. Every rollup then sums **all** node `CostUSD`: `daemon/sessions.go:237-239` (`summarizeGraphs`, backing `catacomb runs`), `daemon/sessions.go:367-369` (`summarizeSession`), `aggregate/aggregate.go:365` (`runTotals`), `aggregate/aggregate.go:153` (`foldRunSteps`), `aggregate/aggregate.go:229` (`foldRunPhases`). Result: total = reported + Σ estimated ≈ 1.68× (V-1: $0.7482 → $1.2560). The same pseudo-node's cumulative `tokens_in/out` also double token sums (`daemon/sessions.go:231-235`, `aggregate/aggregate.go:366-367`) ≈ 2×, and it leaks into any phase window it falls inside.

**Fix (reported-over-estimated precedence, per existing provenance semantics).** Tag the `result`-derived observation as a session-total at capture (`streamjson` attr, copied onto the node the way `model` is at `reduce/reduce.go:79-84`; today `cost_source == "reported"` occurs only on this node — otel maps no cost attribute — but the explicit tag stays correct with a nil pricer and future per-turn reported sources). Rollups then: scope cost = the session-total node's reported value when present, else Σ estimated per-turn (mirroring the reported>estimated rank already used for the `CostSource` label at `daemon/sessions.go:240-251`); token sums and the step/phase window folds exclude the session-total node entirely (per-turn usage remains the attributable source). Regress verdicts were never affected (both sides inflate equally) — this PR fixes absolute reporting; add a live-verify comparing catacomb-recorded vs claude-reported cost on one real `-p` run.

## PR-P — stream-json `result` finalizes the run (F6)

**Mechanism.** Exactly two run-end producers exist: the hook `SessionEnd` (`ingest/hook/hook.go:79-80` → `reduce/reduce.go:40-52`: run `ok`, `EndedAt`, session node closed, statuses cascaded) and the idle reaper's synthetic `run_ended{reason:timeout}` (`daemon/daemon.go:809-838`, 30 m window per `daemon/daemon.go:37` → `applyRunEnded` at `reduce/reduce.go:607-619`: `abandoned`). The stream-json `result` event — the protocol's own terminal record — produces only the cost pseudo-turn (`streamjson.go:135-146`). A hook-less `catacomb run … -- claude -p …` therefore stays `running` for 30 m, ends `abandoned`, has empty `EndedAt`, and contributes no run-scope `duration_ms` samples.

**Fix.** The `result` event is the finalization signal (ADR-0012 §4: `session_ended` fires when a session ends — for a `-p` session, `result` *is* the end). `streamjson` emits an additional `session_end` partial from `result`, `reason` mapped from the result subtype (`success`, `error_max_turns`, `error_max_budget_usd`, `error_during_execution`), riding the existing reducer path unchanged. Interplay guarantees: **no double-end** — `session_end` is idempotent under the reducer, and end-timestamp precedence (`sourceRank` at `reduce/reduce.go:410-424`: hook 2 > stream-json 0) keeps hook timing authoritative when both fire; **no reawaken breakage** — `ensureRun` (`reduce/reduce.go:572-576`) reawakens only `abandoned`, never `ok`, and the reaper skips non-`running` runs (`daemon/daemon.go:814`); the reaper remains the fallback for streams that die before `result`. Docs: the F6-recommended line in `docs/guide/cli.md#bench` flips from "install hooks or lose finalization" to "hooks add timing precision; finalization now works hook-less". Live-verify: a hook-less run must show `EndedAt` set and status `ok` immediately after exit.

## PR-O — Consume `CATACOMB_RUN_ID` (F9, ADR-0005)

**Mechanism.** ADR-0005 makes env `CATACOMB_RUN_ID` the primary multi-session grouping mechanism with `session_id` as fallback. Today it is export-only: `catacomb run --run-id` sets it in the child env (`cmd/catacomb/streamjson.go:160-161`), bench sets it per cell (`cmd/catacomb/bench.go:156`) — and nothing reads it. The stream forwarder sends only `X-Catacomb-Labels` (`cmd/catacomb/streamjson.go:62-64`), the hook forwarder likewise (`cmd/catacomb/hook.go:23,47-49`), and both parsers key runs by session id (`ingest/hook/hook.go:50` and `ingest/streamjson/streamjson.go:79`: `RunID: e.SessionID`). The flag is a no-op; the bench manifest's `run_id` never appears in the store.

**Fix (minimal consumption).** (1) Forwarders send `X-Catacomb-Run-ID` alongside the labels header: `catacomb hook` reads `CATACOMB_RUN_ID` from its own env (inherited: `catacomb run` → claude → hook process), `catacomb run`/bench pass their `--run-id`/cell id directly, `catacomb ingest stream-json` reads the env. (2) Daemon ingest paths thread the value through `IngestWithLabels`/`IngestStreamJSONWithLabels` and override the parsers' `RunID: e.SessionID` default when non-empty. (3) Run forest holds as designed: `ensureRun` already accumulates `SessionIDs[]` (`reduce/reduce.go:580`) and ADR-0011's execution-scoped node ids make cross-session grouping collision-free. (4) Grouped-run lifecycle — `run_id != session_id` becomes reachable for the first time (the branch deliberately deferred in `docs/plans/2026-06-21-m0.3a-graph-finalization.md:905`): per the m0.3 design (`docs/specs/2026-06-21-m0.3-graph-finalization-design.md:118-121`) `session_end` then finalizes only that session, not the run (guard in `reduce/reduce.go:40-52`); the run finalizes when the wrapper exits — `catacomb run` forwards an explicit run-end on child exit (reason `wrapper_exit`, terminal `ok`, requiring a reason-aware branch in `applyRunEnded`, which today hard-codes `abandoned`) — with the idle reaper as fallback (ADR-0012 §4/§5 rejected deriving run end from the last `SessionEnd`). jsonl/otel paths are untouched (transcripts and spans carry no env). Bench selectors stay label-driven; the win is that `runs --run-id`/manifest `run_id` becomes real. If scope pressure appears, (4) may not be cut — it is what makes (1)–(3) safe — but the wrapper-exit signal may fall back to reaper-only with a documented `abandoned` caveat.

## PR-R — `catacomb mcp` stdio server (F8)

**Mechanism.** `docs/guide/concepts.md:73`, `workflows.md:176-181,363`, and `cli.md` document the `mcp__catacomb__mark` convention; the reducer recognizes the tool name (`reduce/marker.go:23-25`) and synthesizes markers from the **tool-call input** riding the trace stream (`reduce/marker.go:34-51`), so ingestion needs zero wiring — but catacomb ships no MCP server, so the tool does not exist client-side and every adopter hand-writes a stub (V-1 §1 did exactly that).

**Fix.** New `catacomb mcp` subcommand: a stdio JSON-RPC 2.0 server, stdlib-only (`encoding/json` + line-delimited stdin/stdout), handling `initialize` (echo protocol version, `capabilities.tools`, `serverInfo{name: "catacomb"}`), `notifications/initialized` (no reply), `tools/list` (one tool `mark`), `tools/call` for `mark` (reply `content: [{type:"text", text:"ok"}]`), unknown method → `-32601`. The `mark` input schema mirrors `markerToolInput` (`reduce/marker.go:27-32`) exactly: `name` string (required), `boundary` string enum `start|end` (required), `occurrence` integer (optional), `state_ref` string (optional); invalid input replies `isError: true` without crashing the session. The server never contacts the daemon — the marker is captured from the stream/hook trace, so it works with zero configuration and fails open. Docs: `--mcp-config` example in `workflows.md` (`{"mcpServers":{"catacomb":{"command":"catacomb","args":["mcp"]}}}`), a `cli.md` section, and a bench-guide note matching the V-1 setup (`--strict-mcp-config`). Live-verify: one real `claude -p … --mcp-config` cell producing a closed phase marker in the graph.

## PR-S — Regress NAME column + doc tails (F5)

**Mechanism.** `regress/render.go:20-25` prints the opaque `KEY` hash per finding while the JSON already carries `name` (`regress/regress.go:184,199`, populated via `rowName` at `regress/regress.go:264-270`) — V-1 needed three manual graph lookups to decode phases. Separately, `AGENTS.md:14` claims unconditionally that "the same observations, in any order, produce the same graph", which overstates the spec: commutativity is per-field given precedence (spec `docs/specs/2026-06-20-catacomb-design.md:233`), status convergence comes from the lattice (spec :244), and inferred statuses stay reversible only until eviction (spec :206, :255).

**Fix.** Add a NAME column to the human regress table (dash when empty; update goldens). Reword `AGENTS.md:14` in one line to match the spec's scoped claim — e.g. "the same observations, in any order, converge to the same graph: per-field precedence makes merge commutative, the status lattice keeps terminals order-independent, and inferred statuses are reversible until eviction (spec §7–§8, ADR-0012/0014)". markdownlint is the docs gate.
